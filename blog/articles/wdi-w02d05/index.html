Midway through second project

Written by Wei

I’m writing this on Sat morning, as a sort of update on what I achieved yesterday and as a sort of work-in-progress review of the things I’m going to need to do today as well.

Yesterday I finished generating collision functions on my game, and made it such that the player can’t move outside of the play field. One major problem with collision is that it’s an all-or-nothing response, and that one collision quickly becomes 4-6 collisions because it detects at each frame. Therefore if I make the player lose a life on every detected collision, a player can lose 5-7 lives on a single move through a bullet. To fix this I added collision thresholds, where a player must accumulate 2-3 collisions to be considered hit, and lose a life. This fixed my problem, until I realised that it was still happening where players would now lose 2-3 lives on one move through a bullet, which is reduced but still not ideal. I thus added a clause where if a player took damage, her collision counter would reset to a larger value than the base, such that she would have to have more collisions after a life loss to trigger a 2nd life loss.

I thought I fixed it there, but then someone pointed out that damage thresholds seemed to be cumulative, where you would be fairly safe after taking the first hit of damage, and have to accumulate collisions to take life damage. I then added two more functions to fix this— basically pushing the player-state towards a base collision threshold value, so that after taking damage the threshold returns to normal after some time, and that if you do make a minor collision that doesn’t reach damage threshold, playing safe causes your damage threshold to return to normal values.

The other major breakthroughs I had yesterday were enemy generation and enemy animation. Based on the position and size of the play area, I can generate enemies that move directly up or down. One problem that stumped me, and now I’m not sure why, is that I couldn’t figure out how to create an array that stored all the enemies I created. For some reason I was convinced it was really difficult and I’d have to use prototypical inheritance and constructor functions, and create objects within objects with unique names, and my collision detection would have to know all these names and somehow have them etc etc etc etc. While all those statements -could- be true, talking with one of the TAs made me realise… I can just use an index as an identifier, and use an array to store my enemies. At this point the problem disappeared and I managed to get my enemy generation working.

The length and complexity of my code is getting to the point where I think I should have created object modules to handle things and separate a whole bunch of things. For example I think my collision calculations should be just one section, my enemy generation another section, and potentially 2-3 more sections. It’s not going to hinder my project, but I really feel like it should be organised a bit better. It’s rather messy right now.

Now things to do:

Important things- Add a game over function, design the loss. Add a game success function, design the success. Add levels (perhaps design difficulty?) Game start function

Added functionality- Generate random sizes of enemies Generate/remove player functions- allowing pauses between stages.

Fun- second screen. hidden scores for second screen. animation for title etc. story.
